開発環境制約（CLI制限）
■ Node.jsコマンド実行禁止
- node
- npx
- npm
- yarn
- bun

理由：
ローカル環境依存の挙動や即時実行による状態改変を防ぐため。
ゲームロジックは純粋なブラウザ実行環境で検証すること。

■ CLIテキスト検索禁止
- grep
- findstr
- sed
- awk
- ripgrep (rg)
- その他コマンドライン置換系ツール

理由：
大量一括置換によるロジック破壊を防ぐため。
検索は必ずエディタの検索機能を使用する。

■ DevTools Consoleでの直接コード実行禁止
- 関数の強制実行
- 変数の直接書き換え
- オブジェクト状態の改変

許可：
- エラーログ確認
- デバッグログ確認

理由：
再現不能な状態汚染を防止するため。

🔎 追加推奨
■ 置換前は必ず影響範囲を確認する
- 参照元ファイル一覧を確認
- 呼び出し元を確認
- SSOT違反にならないか確認

🔐 最重要
■ 「とりあえず直す」は禁止
- 修正前に責任箇所を特定する
- 単一責任の原則を守る

1️⃣ Enemy Status SSOT（Single Source of Truth）
■ Enemyステータスは必ず ENEMY_STATS のみで定義する。

- HP
- Speed
- KnockbackResistance
- CollisionRadius
- SpecialParams

Enemy.js や SpawnDirector 内で直接数値を定義してはならない。
スポーン時の上書き禁止。
倍率処理は base値を元に行う。

2️⃣ スポーン責任の分離
■ SpawnDirectorは「出現させる責任」のみを持つ。

- 敵の能力値を変更しない
- 敵の挙動を変更しない
- ステータスを書き換えない

SpawnDirectorは type を渡すだけ。

3️⃣ フェーズ制の制約
■ フェーズ生成は「構成」を決めるのみ。

- HardCapを超えない
- 役割上限を守る
- 子機(SPLITTER_CHILD)を独立抽選しない
- ペア敵は必ずペア単位で管理

4️⃣ Object Poolの初期化ルール
■ 再利用時は必ず reset() を通す。

- id再発行
- hp初期化
- speed初期化
- フラグ初期化

前回の状態を引き継がない。

5️⃣ Debug Stageの永続化
■ Debug Stageは削除しない。

- 新敵追加時は必ずDebugで検証
- バランス変更はDebugで確認後に反映

6️⃣ 武器バランス原則
■ 武器はDPS差で差別化しない。

差別化は
- 有効状況
- 貫通
- 範囲
- ノックバック

で行う。

7️⃣ 画面外削除ルール
■ OOB削除は即時削除しない。

- マージン有り
- 猶予フレーム有り
- 理由ログ必須

8️⃣ 禁止事項
× 敵ステータスの複数箇所定義
× 子機の独立出現
× HardCap無視のスポーン
× 暗黙の倍率上書き
× デバッグを経ないバランス調整
